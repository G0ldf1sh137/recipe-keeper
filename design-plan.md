# LemmeCook — Design Plan

## Overview
LemmeCook is a web app for creating, organizing, and sharing recipes. Users can write up recipes with ingredients and steps, organize them into collections (presented to users as "cookbooks"), and share them publicly or with specific people via a link.

## Goals
- Let a user quickly capture a recipe (title, ingredients, steps, photo, tags).
- Let a user share a single recipe or a collection via a public link.
- Let other users discover and save (fork/bookmark) recipes shared with them.
- Let a user build a grocery list from one or more recipes, with duplicate ingredients combined.
- Keep the initial scope small: no nutrition calculators in v1.

## Tech Stack
- **Framework**: TanStack Start (React + TanStack Router, full-stack, SSR)
- **Language**: TypeScript
- **Styling**: Tailwind CSS
- **Data fetching/mutations**: TanStack Query + Start server functions
- **Database**: Postgres (via Drizzle ORM); local dev via `docker-compose.yml`, prod via managed Neon Postgres
- **Auth**: Google OAuth sign-in, server-side sessions (HttpOnly cookie, session id hashed at rest)
- **AI**: Claude API (claude-opus-4-8, vision/document input + structured outputs) — transcribes a handwritten recipe from uploaded photos or an uploaded PDF into the recipe record via an owner-only "Process photos"/"Process PDF" button with preview-then-confirm; a web-scraping importer (fetches a recipe's source URL via the `web_fetch` tool and extracts title/ingredients/steps/photos) is built but not yet wired into the UI
- **Deployment target**: Vercel (via the Nitro Vite plugin) + Neon Postgres. Recipe photos stored in S3.
- **PDF export**: `@react-pdf/renderer` (pure-JS PDF renderer, no headless browser) generates a downloadable PDF of a recipe on demand, server-side.

## Core Data Model
- **User**: id, email, name (editable), username (unique, auto-generated on signup, editable), avatarUrl (from Google OAuth profile), avatarOverrideUrl (nullable, a user-uploaded custom avatar that takes precedence over avatarUrl when set), createdAt
- **Recipe**: id, ownerId, parentRecipeId (nullable, self-referencing — set when this recipe was forked from another, cleared to null if the original is later deleted), title, description, ingredients (list of {qty, unit, name}), steps (ordered list of {text, imageUrls}), photoUrls (list of strings), coverPhotoUrl (nullable, must be one of photoUrls, shown on recipe cards), sourceUrl (nullable, link to the recipe's original source), sourcePdfUrl (nullable, S3-hosted PDF the recipe was imported from — kept as a persisted attachment, shown as a download link on the detail page), tags (list of strings), yield (free text, e.g. "4 servings", nullable), calories (per serving, integer, nullable), visibility (private | public, defaults to public for new recipes), createdAt, updatedAt
- **Collection**: id, ownerId, name, description, visibility
- **CollectionRecipe**: collectionId, recipeId (join table)
- **Calendar**: id, ownerId, name, visibility, createdAt, updatedAt — a reusable weekly meal-plan template (Mon–Sun slots, not tied to a specific date)
- **CalendarEntry**: id, calendarId, recipeId, dayOfWeek (mon–sun), createdAt — a recipe assigned to a day; the same recipe can appear on multiple days or twice on one day
- **Share**: id, recipeId or collectionId or calendarId, token (for share links to public resources), createdBy, createdAt
- **Comment**: id, recipeId, authorId, parentId (nullable, self-referencing — threaded replies), body, createdAt, updatedAt
- **Rating**: recipeId, userId (composite key — one rating per user per recipe), value (1–5), createdAt, updatedAt
- **GroceryList**: id, ownerId, name, createdAt, updatedAt
- **GroceryListItem**: id, listId, recipeId (nullable — null means manually added), qty, unit, name, checked, createdAt
- **Ingredient/Unit reference tables**: global `ingredients` and `units` tables of distinct names, grown automatically as recipes are saved, powering autocomplete on the ingredient-name and unit fields

## Key Screens / Routes
- `/` — Landing/feed: public recipes + user's own recent recipes if logged in
- `/about` — Static, curated feature summary; public, no login required
- `/login`, `/signup` — Auth
- `/recipes/new` — Create recipe form
- `/recipes/$recipeId` — View a recipe (ingredients, steps, photo, author, share button); a 0.5x/1x/2x/custom toggle scales the displayed ingredient quantities (display-only, nothing persisted)
- `/recipes/$recipeId/edit` — Edit recipe (owner only)
- `/recipes/$recipeId/pdf` — Downloads a generated PDF of the recipe (same visibility rules as the recipe page; not a navigable page)
- `/collections` — List user's collections ("Your cookbooks" in the UI)
- `/collections/$collectionId` — View a collection and its recipes (owner controls, plus anonymous/shared viewing)
- `/calendars` — List user's Meal Weeks (weekly meal-plan calendars internally; "Your Meal Weeks" in the UI)
- `/calendars/$calendarId` — View a Meal Week as a 7-day grid, add/remove/reorder recipes per day (owner controls, plus anonymous/shared viewing); any viewer who can see it can bulk-add every scheduled recipe's ingredients into one of their own grocery lists in one click
- `/u/$username` — Public profile: a user's public recipes and public lists
- `/settings` — Change your username
- `/shared/$token` — Resolves a share link and redirects to the canonical recipe/collection/calendar page (no login required)
- `/grocery` — List user's grocery lists
- `/grocery/$listId` — View a grocery list: recipes' ingredients combined, manual items, check off while shopping

## Sharing Model (v1) — done
- **Public**: directly viewable by anyone at its normal URL, listed in `/recipes` browse. New recipes default to public; collections and calendars default to private.
- **Private**: visible only to the owner; not shareable.
- (An earlier "Unlisted" middle tier — visible only via a share link, not listed anywhere — was removed to keep the model to just private/public.)
- Sharing a recipe, collection, or calendar generates (or reuses) a `Share` record with a random token, one per resource; revoking deletes the row. Visibility is re-checked live on every access, so setting a resource back to private immediately invalidates any existing share link, without needing to explicitly revoke it first. Since only public resources can be shared, a share link is mostly a convenience/short-link, not a privacy boundary.
- Public profile pages (`/u/$username`) list a user's public recipes and public lists; usernames are auto-generated at signup and editable via `/settings`.

## Non-Goals (v1)
- Automatic nutrition calculation from ingredients (a recipe can optionally store a manually-entered yield and calorie count, but nothing is derived from ingredient data)
- Recipe scaling/unit conversion

## Backlog / Future Ideas
Not yet started — captured here for later prioritization.
- Real self-serve subscriptions ($5/month) for subscriber-only features (milestone 46's grocery/pantry/calendars gate, plus AI import from milestone 34) — would need a payment provider (e.g. Stripe) wired to the existing `users.isSubscriber` flag, replacing today's admin-only manual toggle.
- Collapse forked recipes together in search/browse results — today `findForksOfRecipe`/`parentRecipeId` (`recipes.server.ts`) only surface the fork relationship on a recipe's own detail page ("Forked from..."); the recipes browse page lists an original and all its forks as flat, separate entries.
- Add OAuth sign-in via Facebook, X, and Pinterest, alongside the existing Google OAuth flow (`src/auth/google.server.ts` + the `/auth/google` routes) — would need a provider-specific server module and callback route per network, plus a way to link/merge multiple provider identities to one `users` row.

## Milestones
1. Scaffold app (TanStack Start + Tailwind), basic routing shell
2. Data layer: Drizzle schema + migrations, server functions for CRUD
3. Recipe CRUD UI (create/view/edit/delete)
4. Auth (signup/login/session)
5. Collections (create, add/remove recipes)
6. Sharing — done: revocable share links for public recipes and collections, plus public profile pages (`/u/$username`).
7. Grocery lists — done: multiple named lists, add/remove a recipe's ingredients with duplicates combined (quantities are parsed as exact fractions, so whole numbers, decimals, simple fractions like "1/2", and mixed numbers like "1 1/2" all sum correctly, e.g. 1/2 + 1/4 = 3/4), manual items, check off; ingredient-name and unit autocomplete on the recipe form.
8. Recipe forking — done: clone any recipe you can see into your own private copy, referencing the original; the original's page lists its forks (visible ones only).
9. Polish — done: exact tag/visibility filters plus free-text search (title/description) on `/recipes`, with lightweight client-side search on collections and profile pages; responsive layout (mobile nav, wrapping rows, responsive grids/padding) across the app; empty-state messaging on every list view.
10. Calendars — reusable weekly meal-plan templates (Mon–Sun slots, multiple recipes per day), shareable like collections via public/private visibility plus revocable links.
11. Recipe import — done: transcribe a recipe from an uploaded PDF via the same Claude pipeline used for handwritten photos, with the PDF kept as a persisted, downloadable attachment on the recipe. A `sourceUrl` field records a recipe's origin link (shown on the detail page). (The original web-scraping importer here used Claude's own `web_fetch` tool and was blocked by most sites' bot detection — superseded by milestone 37's structured-data approach.)
12. Recipe export — done: a "Print to PDF" button on the recipe detail page generates a real, downloadable PDF (title, ingredients, steps, photos, source attribution) server-side via `@react-pdf/renderer`, respecting the same public/private visibility rules as the page itself.
13. Visibility simplification — done: removed the "unlisted" tier, leaving just private/public across recipes, collections, and calendars; new recipes now default to public (collections/calendars still default to private).
14. Recipe scaling — done: a 0.5x/1x/2x/custom toggle on the recipe detail page scales displayed ingredient quantities on the fly (page-local, not persisted). The fraction-parsing math from milestone 7 was extracted into a shared `src/recipes/quantity.ts` module so both the server-side grocery summing and this client-side scaling reuse the same exact-rational arithmetic; a custom factor field accepts whole numbers, decimals, or fractions (e.g. "3/4"). Non-numeric quantities (e.g. "pinch") are left unchanged at any scale.
15. Calendar-to-grocery-list bulk add — done: a widget on the calendar page adds every scheduled recipe's ingredients to a chosen (or new) grocery list in one click, available to any viewer who can see the calendar (owner or via share link), not just the owner. Ingredients are added once per scheduled instance (not deduped by recipe), so a recipe planned on 3 different days contributes its ingredients 3 times — the existing quantity-summing logic then correctly totals the week's shopping list. Not idempotent by design: re-clicking adds everything again.
16. Calendar entry reordering — done: recipes within a single calendar day can be reordered (owner only) via ↑/↓ buttons next to the existing remove button, backed by a `position` column on `calendarEntries` and a neighbor-swap update (no drag-and-drop dependency — the first manual-ordering feature in the app). New entries append after the current max position for that day; removing an entry leaves a harmless gap since ordering only needs relative order, not contiguity.
17. Paste-to-import + new-recipe chooser — done: `/recipes/new` now opens with an upfront choice — import from photo, import from PDF, import from pasted text, or start from a blank form — instead of always showing the full form. Paste-to-import is a new capability reusing the same Claude transcription pipeline as photos/PDF (`transcribeRecipeText`, same shared schema and prompt/preview/apply conventions); each import screen has "← Back" and "Skip, fill in manually" links. Only the photo importer stays visible once you reach the full form (to add more photos to transcribe); the PDF and paste-text importers are reachable only through their dedicated chooser screens, not shown again inline on the form. `/recipes/$recipeId/edit` keeps only the photo importer above the form for the same reason — editing already has content, and there's no chooser there.
18. Notifications — done: a recipe owner gets an in-app notification when someone else comments on, forks, or rates their recipe (self-actions on your own recipe never notify you). A bell icon in the header (always visible, not tucked behind the mobile hamburger menu) shows an unread-count badge and links to `/notifications`, which lists recent activity newest-first and auto-marks everything read on visit. Re-rating a recipe you've already rated doesn't spam a second notification — only the first rating per user/recipe fires one. Email is out of scope for now (no email-sending infrastructure exists in the app yet); `users.email` is already captured via Google OAuth for whenever that's added.
19. Cookbook PDF export — done: a "Print to PDF" link on a cookbook (collection) page bundles every recipe in it into one PDF — a cover page (name, owner, recipe count, table of contents) followed by one full recipe page per item, reusing the exact same per-recipe layout as the existing single-recipe export (`RecipePdfPage`, extracted from `RecipePdfDocument` so both share one implementation). A recipe the viewer can no longer access (e.g. added while public, later made private by its owner) is silently omitted from the generated PDF rather than erroring or leaking its content — mirrors the fact that its own page wouldn't be viewable either.
20. Extended nutrition info — done: protein/carbs/fat (grams per serving) join calories as recipe fields, filled in by hand or estimated by Claude alongside calories during photo/PDF/text import (and the disabled URL-scraping module, kept in parity though still unwired). Displayed as a second meta line under yield/calories on the recipe detail page and in both PDF exports, omitting any of the three that are unset; like calories, none of the three are affected by the 0.5x/2x/custom scaling toggle.
21. Transcription source URL + ingredient normalization — done: photo/PDF/text import now also extracts a source URL when one is written or shown in the input (printed on a card/PDF, mentioned in pasted text), filling the recipe's existing Source URL field automatically. Separately, transcription now normalizes each ingredient's name and unit to a clean, singular, standard form (e.g. "onions"/"diced onion" → "onion", "tbsp" → "tablespoon") and is given the app-wide list of previously-used ingredient/unit names as a hint to reuse an exact existing spelling when the ingredient is the same thing — so recipes imported at different times combine correctly into one summed line on a shared grocery list instead of staying split by wording differences. Verified end-to-end: two separately-imported recipes sharing "onion" combined into a single "3" line on a shared grocery list.
22. Admin override — done: a new `users.isAdmin` flag (set via a one-off direct DB update — no in-app UI to grant it) lets an admin open any recipe, cookbook, calendar, or grocery list directly by its URL and view, edit, and delete it, bypassing the normal owner/private-visibility check. The bypass is scoped narrowly: browse listings (homepage, `/recipes`, `/collections`, profile pages) are unaffected and still only show public + own content to everyone including admins — the override only kicks in once an admin is looking at one specific item by URL, not a "see everything" moderation dashboard. Share-link creation/revocation stays owner-only even for admins. A "Viewing as admin" banner appears whenever an admin is looking at something they don't own. Comments and ratings on a recipe (which independently re-check recipe visibility before loading/posting) needed the same bypass threaded through — the first end-to-end test caught this: `getRecipe` succeeded but the recipe page still 404'd because `listComments`/`getRatingSummary`, loaded in parallel by the same route, weren't yet admin-aware.
23. Admin panel — done: a `/admin` page (nav-linked only for admins) lists every user with a one-click "Make admin"/"Revoke admin" toggle, replacing the manual DB update milestone 22 shipped with. An admin can't see a toggle on their own row (shows "You" instead) — the server function also rejects a self-targeted change outright — since accidentally revoking your own access with no other admin around would strand the app back at manual-DB-only.
24. Reporting / moderation queue — done: any signed-in user can report a recipe or a comment (a small collapsed "Report" control next to Fork/Reply that expands into a reason field), and open reports surface in a new "Reports" section on `/admin`. Recipe reports resolve by marking them reviewed — the admin uses the existing admin-override edit/delete to act if warranted. Comment reports get real teeth: this also added the first-ever comment deletion (admin-only; there was previously no way to delete a comment at all, not even for its own author), so a comment report can be "Deleted" (removes the comment and its replies, which cascades away the report row too — no separate "resolved" state needed) or "Dismissed" if unfounded. A user can't report their own recipe or comment, and the report control itself is hidden on your own comments.
25. Ingredient-based recipe search — done: `/recipes` gained an "Ingredient" filter alongside the existing Tag/Search/Visibility filters, matching partial/case-insensitive against ingredient names via a `jsonb_array_elements` scan over the `ingredients` column (which stores `{qty, unit, name}` objects, not a native array, so it needed its own query helper rather than reusing the tag filter's `arrayContains`). Paired with a datalist autocomplete sourced from the same `ingredientNames` table used elsewhere, so typing nudges users toward the canonical spelling that transcription normalization (milestone 21) already produces — searching "onion" reliably finds every recipe using it, instead of being defeated by "onions"/"diced onion" wording differences. Combines with the other filters as AND, same as the existing tag + search combination.
26. Tag system redesign — done: tags now go through the same lowercase/trim/dedupe normalization ingredients and units already get (applied directly in the recipe schema via `.transform()`, since tags — unlike ingredients — are always hand-typed with no AI-transcription step to lean on), and a new `tags` reference table (mirroring `ingredientNames`/`unitNames`) grows automatically as recipes are saved, powering a shared chip-editor (`TagInput`) used both on the recipe form and (at the time) the `/recipes` filter, with a live suggestion dropdown as you type. Tag chips are clickable everywhere they're shown (recipe cards, recipe detail page), linking into `/recipes` pre-filtered to that tag — `RecipeCard` needed restructuring since the whole card used to be one link and a tag chip can't be its own link nested inside another. A new `/recipes/tags` page lists every tag in use with a recipe count (respecting visibility — a private recipe's tags don't inflate the count for other viewers) and links into the filtered view. (The multi-tag AND filter UI built here was superseded by milestone 27's single combined search box — tag chips still link out to `/recipes`, just via the unified `q` param now instead of a dedicated `tags` param.)
27. Unified recipe search — done: the separate Search/Tags/Ingredient filters on `/recipes` (added piecemeal across milestones 71, 25, and 26) collapsed into one text box. A single term now matches if it appears in the title or description, OR matches a tag, OR matches an ingredient name — plain OR-across-fields, not the precise multi-tag AND-select the old dedicated Tags chip-filter had. Tag chips (on recipe cards, the detail page, and `/recipes/tags`) link into this box via the `q` param instead of a dedicated `tags` param. Traded the old precision (exact multi-tag AND, exact ingredient match) for a simpler, single-input "search everything" UX matching how most recipe sites' search bars work; the `TagInput` chip-editor and ingredient-name datalist remain in place on the recipe form itself, only the `/recipes` browse filters were consolidated.
28. Similar recipes — done: the recipe detail page gained a "Similar recipes" section, ranking other visible recipes by how many tags and ingredient names they share with the one being viewed (simple overlap count, ties broken by recency), reusing the same `RecipeCard` list styling as the existing "Forked N times" section. Computed inline inside the existing `getRecipe` server function (it already has the recipe's tags/ingredients in hand, so no extra round trip), using a raw SQL scoring query — the first attempt used `= any(...)` with an interpolated array and failed at runtime, because drizzle expands an interpolated JS array into a parenthesized parameter list for `in (...)` use, not a single array-typed parameter; switched to `in` (guarded against empty arrays, which are invalid inside `in ()`) to fix it. Respects the same visibility rule as everything else — a private recipe's shared tags/ingredients never surface it in another viewer's similar-recipes list, verified by creating a private test recipe sharing a tag and ingredient with a public one and confirming it appeared for its owner but not for an anonymous viewer. A recipe with no overlap with anything shows no section at all, not an empty one.
29. Cook mode — done: a full-screen, one-step-at-a-time view at `/recipes/$recipeId/cook`, reusing `getRecipe` for the same visibility/share-token gating the normal detail page already has (verified: a private recipe's cook-mode route 404s the same way for a non-owner viewer). The standard app header is hidden for this one route (`__root.tsx` checks the path and skips rendering it) for a distraction-free layout. Steps are navigated via large Previous/Next buttons, left/right tap zones, or arrow keys, with a collapsible ingredients panel offering the same 0.5x/1x/2x/custom scaling as the main recipe page — that scaling logic (previously only inline on the detail page) was extracted into a shared `useRecipeScale` hook + `ScaleToggle` component now used by both. Requests a screen Wake Lock on mount so the display doesn't sleep mid-recipe, feature-detected and wrapped in a try/catch since the request can be unsupported or denied depending on browser/environment (confirmed this exact path during verification: the API was present but denied permission in the test environment, and cook mode kept working normally).
30. Cook mode timers — done: steps with a detectable duration ("Bake at 350 for 10-12 minutes.") get a "Start N:00 timer" button (`parseStepDuration`, a small regex over the step text — a range like "10-12" uses the larger number so the timer errs toward "not done yet" rather than under-cooking). Timer state lives at the cook-mode page level rather than per-step, since starting a bake timer and then reading ahead to the next step is the whole point — a persistent countdown pill in the header keeps ticking and stays visible across step navigation until canceled or dismissed. One timer at a time; starting a second replaces the first. On completion it plays a short tone (Web Audio `OscillatorNode`, no external asset — the `AudioContext` is created/resumed at Start-timer click time to satisfy the browser's audio-needs-a-user-gesture rule, then reused later when the countdown actually reaches zero) and vibrates on supported devices, both feature-detected the same way the Wake Lock check in cook mode already is. Verified end-to-end with a temporary short-duration test step: countdown ticks, survives step navigation, a second timer replaces the first, and completion correctly shows "Time's up" with a working Dismiss.
31. Public cookbook discovery — done: a new `/collections/browse` page lists every public cookbook across all users (plus your own regardless of visibility, the same "public or own" rule used everywhere else), searchable by name, each showing its owner and recipe count, most recent first — closing the gap where `/collections` only ever showed *your own* cookbooks and there was no way to stumble onto someone else's without already having a direct link. Reuses the existing `collections`/`collectionRecipes`/`users` tables (no migration) via a new global `findPublicCollections` alongside the existing per-owner `findPublicCollectionsByOwner`. Linked from `/collections` via a "Browse public cookbooks" line, same low-key entry-point pattern as `/recipes`'s "Browse tags" link rather than a new top-level nav item. Verified visibility end-to-end: a private test cookbook showed up (with a "Private" badge) only in its owner's own view of the browse page, not for an anonymous or other-user viewer.
32. Personal recipe notes — done: a private per-user "My notes" textarea on the recipe detail page, for jotting things like "used less salt, double the garlic" without it becoming a public comment — every other way of interacting with a recipe (comments, ratings, saving) is visible to at least the recipe's owner, this is the first fully private layer. New `recipe_notes` table mirrors the exact one-row-per-(recipe,user) shape `ratings` already uses (composite primary key on `(recipe_id, user_id)`, no surrogate `id`), with one deliberate difference: "no note" is modeled as "no row" — saving an empty (trimmed) textarea deletes the row rather than storing an empty string, so there's no separate delete button. `getMyNote`/`saveMyNote` are fully `requireAuthMiddleware`-gated with no anonymous read path at all (unlike ratings, which allow anonymous aggregate viewing), and are always scoped to `context.user.id` — a deliberate, explicit carve-out from the app-wide admin-override pattern (milestone 22), since notes are private user data rather than owned content, so there's no "view another user's note" path for anyone, including admins. Verified end-to-end: persistence across reload, edit-and-resave produces exactly one row (not a duplicate), clearing the textarea and saving deletes the row, a note is invisible to any other signed-in user viewing the same recipe, signed-out visitors see a "Sign in to add a note" prompt with no content leak, and deleting the underlying recipe cascades away its notes too.
33. Recipe-detail "add to" widgets redesign — done: `SaveToList`, `AddToGroceryList`, and `AddToCalendar` were each a full-width, always-open section with a chip row and a permanently-visible "create new" form — three repetitive sections of dead vertical space on every recipe page. Replaced with a new shared `src/ui/DropdownButton.tsx` primitive (the app's first anchored popover — extends the existing "boolean state + `absolute top-full` panel" convention already used by the header's mobile nav, adding outside-click and Escape dismissal, neither of which existed anywhere in the app before): each widget now renders as a single compact trigger button with an at-a-glance badge count, opening a small panel on click. The "create new" input collapses to a "+ New..." affordance that only expands on demand (same convention as `src/reports/ReportButton.tsx`), and `AddToCalendar`'s old two-step "pick a day from a `<select>`, then click a separate + Add button" flow was replaced with 7 single-click Mon–Sun toggle pills per calendar — matching the same one-click toggle model the other two widgets already used, so all three widgets now share one consistent interaction. Creating a new calendar is now name-only (no day picker at creation), a small deliberate behavior change traded for consistency. Verified end-to-end: toggling chips/pills adds and removes without a full reload, badges update live, outside-click and Escape both dismiss an open panel, opening a second dropdown correctly closes whichever was already open (confirmed with real click events — synthetic `.click()` calls don't fire `mousedown` and so don't exercise the outside-click listener, a distinction worth remembering for future popover testing), the signed-out view shows plain sign-in links with no dropdown affordance, and the layout wraps cleanly with no panel overflow at mobile width.
34. Per-user AI transcription access flag — done: a new `canTranscribe` boolean on `users` (default `false` — opt-in, so the migration itself revoked every existing user's transcription access until an admin explicitly grants it, same as a brand-new signup), toggleable per-user from a second "Enable/Disable AI import" button on each `/admin` user row (mirrors the existing "Make/Revoke admin" control exactly — `setUserCanTranscribe` copies `setUserAdmin`'s shape in `admin.functions.ts`). Enforced by a new `requireTranscriptionAccessMiddleware` (`auth-middleware.ts`, identical shape to `requireAdminMiddleware`) swapped in for `requireAuthMiddleware` on all three transcription server functions (`processRecipePhotos`/`processRecipePdf`/`processRecipeText`) — admins always bypass the flag, same app-wide admin-override pattern used elsewhere. `ProcessPhotos`/`ProcessPdf`/`ProcessText` take a `canUse` prop that disables their action button and swaps in an explanatory message when false; `/recipes/new`'s three import-chooser buttons and `/recipes/$recipeId/edit`'s `ProcessPhotos` render are gated the same way (both routes now thread the session user through their loader to compute `isAdmin || canTranscribe`). Verified end-to-end: a fresh non-admin user starts with every AI tool disabled in the UI; flipping the DB flag off *mid-session*, without reloading the page, causes the still-client-enabled "Process text" button's request to fail server-side — proving the UI disabling is UX only and the middleware is the real boundary; an admin with their own flag off can still use every tool; the admin panel's own "Enable/Disable AI import" button correctly flips access live, no full reload needed.
35. `/about` page — done: a new static, public route (no auth) summarizing the app's feature set for a visitor who hasn't signed in yet — previously `/` redirected straight to `/recipes` and a signed-out header showed nothing but "Sign in with Google", so there was no way to learn what the app does before creating an account. Content is a curated list of 12 user-facing feature groups (recipe basics, AI import, cook mode, scaling, cookbooks & meal planning, grocery lists, search & discovery, sharing & visibility, community, personal notes, export, dark mode) written in plain language, deliberately translated from — not copy-pasted out of — this file's more technical milestone prose. Linked from a new "About" header link in `src/routes/__root.tsx`, placed unconditionally (outside the existing `user ? ... : ...` branch) so it's the first thing a signed-out visitor can click. **Standing habit going forward**: whenever a new feature is user-facing enough to earn a milestone here, also add or update a line on `/about` in the same pass — this list is not a one-time snapshot.
36. Notification preferences — done: `/settings` gained a "Notifications" section with three checkboxes (comments/ratings/forks on your recipes), each defaulting to checked (`notifyOnComment`/`notifyOnRating`/`notifyOnFork` on `users`, `notNull().default(true)` — opt-out, not opt-in, since existing users already receive every notification today and nothing should regress silently; the opposite default from milestone 34's `canTranscribe`, which was deliberately opt-in for a cost/abuse control rather than a UX preference). Enforcement lives in exactly one place — `insertNotification` (`src/notifications/notifications.server.ts`) — which now looks up the recipient's preference for the notification's type and skips the insert if disabled, alongside its existing no-self-notification guard; none of the 3 call sites (`createComment`, `rateRecipe`, `forkRecipe`) needed any change.
